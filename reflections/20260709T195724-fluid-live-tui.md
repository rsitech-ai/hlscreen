# Fluid live TUI pass

## Success criteria

- `hls tui` launches the live workstation with top-10, color, interactive TUI defaults.
- The live TUI does not rebuild and clear a terminal on every progress tick.
- Existing read-only/live recording safety remains unchanged.
- Focused tests cover the CLI shortcut and persistent render sink behavior.

## Failure hypotheses

1. Flicker is caused by `Terminal::new` plus `terminal.clear()` on every progress frame.
2. Reusing `LiveArgs` directly for a shortcut would make top-10 defaults ambiguous when a user explicitly passes `--top 50`.
3. Passing a mutable renderer through the WebSocket loop can create borrow conflicts around interactive redraws.

## Candidate approaches

- Add a persistent renderer object created once after terminal mode setup, then pass a mutable draw sink through progress rendering.
- Add a dedicated `TuiArgs` type that converts into `LiveArgs` with workstation defaults instead of mutating `LiveArgs` after parsing.
